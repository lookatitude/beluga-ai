# Design Context — Beluga AI v2

This file persists design context for all frontend work on the Beluga AI documentation and marketing site (`docs/website/`). Loaded by the `frontend-design` skill and related tools before any design task.

## Project

**Beluga AI v2** — Go-native agentic AI framework (`github.com/lookatitude/beluga-ai/v2`). Go 1.23+, streaming via `iter.Seq2[T, error]`, 110+ providers, 7-layer architecture, production-first.

**Site stack:** Astro 5 + Starlight + Tailwind v4 + React 19 + MDX + Mermaid. Dark default, light mode supported. Lives in `docs/website/`.

**Surfaces covered:** marketing (`index`, `about`, `compare`, `enterprise`, `integrations`, `community`, `features`) + full Starlight docs tree.

## Design Context

### Users

Three overlapping audiences, all served by the same site:

1. **Senior Go backend engineers** evaluating agent frameworks against LangChain/LangGraph/Mastra. They read code first, marketing copy second. They want to see `iter.Seq2`, real interface definitions, real benchmarks. They're allergic to hand-waving.
2. **Platform / infra leads** building internal AI tooling for their org. They care about observability (OTel GenAI spans), resilience (circuit breakers, rate limits), durability (workflow), deployment modes (Docker, K8s, Temporal, embedded), and auditability.
3. **Go application developers** new to agent frameworks. They need the 5-minute first-agent quickstart to actually work, and a clear mental model of the 7 layers before they commit.

**Shared job-to-be-done:** "Convince me in under 10 minutes that this is the Go-native agent framework I should bet production on — then get out of my way so I can read the code."

### Brand Personality

**Three words:** rigorous · editorial · unflinching.

Supporting axes:
- **Go-native, not Python-ported.** The site should feel like it was designed by people who ship Go in production — not a JS team marketing a Python framework in disguise.
- **Production-serious.** Observability, resilience, and durability are default concerns, not afterthoughts. The design should reflect that weight.
- **Technically honest.** Show the interface. Show the file:line. Show the tradeoff. Never dress up vapor.

**Emotional goals (layered):**
- *Confidence* — "this is built by people who know Go and agents deeply."
- *Calm / trust* — "boring, reliable infrastructure I can build a business on."
- *Excitement* — "finally, something fresh in a space dominated by Python DAG toys."

### Aesthetic Direction

**Tone:** editorial-technical. Think a technical trade publication crossed with Vercel's precision and CrewAI's confidence — but rigorously Go-flavored. Generous whitespace, strong typographic hierarchy, asymmetric compositions, purposeful decorative marks (not glassmorphism, not gradient blobs).

**References (the feel we want):**
- **Vercel.com** — typographic confidence, monochrome foundation with precise accent use, generous space, motion that serves meaning.
- **CrewAI.com** — clean product narrative, strong hero, honest capability grid.

**Anti-references (what to avoid at all costs):**
- Cyan-on-dark with purple-to-blue gradients, neon glow borders, glassmorphism everywhere.
- Identical icon-card grids (icon + heading + 3 lines, repeated 6-9 times).
- "Big hero metric + supporting stats + gradient accent" template.
- Inter everywhere, rounded rectangles with generic drop shadows, centered everything.
- Marketing language without code. Every claim must point to a file:line or a working example.

**Theme:** dark default (keep it), light mode supported. Never pure black (`#000`) or pure white (`#fff`) — always tinted toward the primary hue.

**Color palette (anchored to the logo — this is the one hard constraint):**
- **Primary:** `#5CA3CA` (steel blue, from logo). This is load-bearing; do not replace.
- **Primary accent:** `#C9E1E5` (pale aqua, from logo). For highlights, hover states, decorative marks.
- **Neutrals:** tint toward the primary hue. Background dark is a cool near-black (`oklch` close to `#0b0e11`), not neutral gray. Light mode background is a warm near-white tinted with a breath of aqua.
- **Accent (sparingly):** a deeper teal or a contrasting warm for CTAs and emphasis — chosen to sit in the same OKLCH lightness band as the primary.
- **Forbidden:** purple-to-blue gradients, neon cyan, rainbow accents. The logo palette is cool and restrained; keep it that way.

**Typography (free to change — Inter is out):**
- **Display / headings:** a distinctive, editorial sans or a precise geometric — candidates: *Söhne*, *GT America Mono* for select marks, *Neue Haas Grotesk*, *General Sans*, *Inter Display* (only if differentiated from body), *Migra* for editorial moments, *Space Grotesk* if a technical-but-warm feel is desired. Final choice commits to one distinctive display face.
- **Body:** a refined, boring-in-the-best-way text face — *IBM Plex Sans*, *Söhne*, *Untitled Sans*, or a variable neo-grotesque. Must read well at 16px on dark backgrounds.
- **Mono:** *JetBrains Mono*, *Berkeley Mono*, or *IBM Plex Mono*. Used for code, file:line refs, and selectively as an editorial mark (e.g. layer numbers, section labels) — not as a lazy "technical vibe" shorthand across the whole site.
- **Scale:** modular, fluid via `clamp()`. Strong contrast between display and body sizes.

**Layout & rhythm:**
- Asymmetric compositions. Break the grid intentionally on marketing pages.
- Varied spacing — tight groupings for dense code, generous separations between conceptual sections.
- Left-aligned by default; center only for deliberate emphasis.
- Container queries (`@container`) for component-level responsiveness.
- Docs sections can be denser and more utilitarian; marketing pages must have editorial breath.

### Design Principles

Every design decision defers to these, in order:

1. **Code is the hero.** Every page that makes a technical claim shows real code — with full imports, real error handling, and a file:line reference. No pseudocode in reference surfaces. Marketing pages must include at least one working `iter.Seq2` snippet within the first viewport whenever possible.
2. **Typographic hierarchy over decorative chrome.** Before adding a card, a shadow, a border, or a background blob, ask whether type size, weight, and spacing could carry the hierarchy instead. They usually can.
3. **Dark-default, tinted, restrained.** Logo primary (`#5CA3CA`) and aqua (`#C9E1E5`) are the only chromatic anchors. Everything else is a tinted neutral. No second accent hue without explicit justification.
4. **Asymmetric, editorial, Go-flavored.** The site should feel like a technical publication edited by someone who ships Go — not a templated SaaS landing page. Break the grid where it earns emphasis.
5. **Every claim earns its pixels.** If a section doesn't advance a user's decision (evaluate, extend, deploy, operate), it doesn't ship. Cut copy that restates the heading. Cut icons that don't convey meaning. Cut stats that don't anchor to a file.

### Accessibility

- WCAG AA minimum for all marketing and docs surfaces. AAA for body copy where achievable.
- Respect `prefers-reduced-motion` on every animation — including the hero.
- Contrast checked in both themes at every token change.
- Keyboard-complete: every interactive surface must be reachable and operable without a pointer. Visible focus rings tinted with the primary.
- Avoid conveying meaning through color alone (especially for the compare page and layer diagrams).

## Notes for future design sessions

- The logo palette is the single hard color constraint. Everything else is negotiable per session.
- Inter is out unless there's an explicit reason to keep it.
- When in doubt between "add something" and "remove something," remove.
- The AI-slop fingerprints in the `frontend-design` skill's DON'T list apply with full force on this project.
